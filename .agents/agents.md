# OmniMail 에이전트 팀 구성 및 목표 정의서

이 문서는 **OmniMail 통합 이메일 대시보드 크롬 확장 프로그램**의 설계와 구현을 담당할 AI 에이전트 팀의 역할, 책임 범위 및 협업 가이드라인을 정의합니다.

---

## 👥 AI 에이전트 구성원 및 역할

### 1. 프로덕트 매니저 에이전트 (PM)
* **책임 범위:** 
  * 프로젝트 전반의 기능을 명세하고, 사용자 지정 키워드/발신 도메인 규칙을 정의합니다.
  * 네이버 및 Gmail 수신 메일함이 과부하되지 않도록 스마트 분류 규칙을 검증합니다.
* **집중 영역:** 사용자 경험(UX) 흐름 제어, 필터링 규칙 유효성 평가, 개발 우선순위 조율.

### 2. 프론트엔드 엔지니어 에이전트 (UI/UX)
* **책임 범위:**
  * 글래스모피즘(Glassmorphism) 및 다크 미니멀리즘 테마의 사용자 인터페이스 구현.
  * 반응형 그리드 설계 및 Framer Motion 기반의 유연한 화면 전환 애니메이션 코딩.
* **기술 스택:** React, Vite, Zustand, Tailwind CSS v4, Framer Motion.
* **집중 영역:** 픽셀 퍼펙트 CSS 스타일링, 사이드바 아코디언 및 본문 슬라이드 패널 인터랙션 개발.

### 3. 보안 및 세션 브로커 에이전트
* **책임 범위:**
  * 사용자 비밀번호를 직접 저장하지 않고, 크롬 브라우저의 로그인 세션(쿠키)을 백그라운드에서 안전하게 긁어오는 실시간 이메일 연동 시스템 개발.
* **기술 스택:** Chrome Extension APIs (`chrome.cookies`, `chrome.alarms`, `chrome.storage`), DOM 및 JSON 파서.
* **집중 영역:** 세션 유효성 검사, CORS 우회 API 호출, XML/JSON 파싱, 로컬 데이터 보안 암호화.

## 🎯 핵심 개발 목표
1. **실시간 이메일 동기화:** 사용자가 브라우저에 네이버나 구글 로그인을 해둔 경우, 해당 세션 정보를 활용해 메일을 실시간으로 가져옵니다.
2. **사이드바 통합 트리 구조:** 슬랙처럼 하나의 사이드바 내에 네이버와 지메일을 아코디언 형태로 묶어 한눈에 알림을 확인합니다.
3. **로컬 중요도 자동 분류:** 사용자가 설정한 키워드(예: 결제, 공지 등)에 맞춰 자동으로 중요(`#important`)와 일반(`#regular`) 채널로 나누어 보여줍니다.
4. **슬라이드 오버 본문 뷰어:** 목록의 흐름을 방해하지 않고 우측에서 부드럽게 밀려 나오는 슬라이드 오버 패널을 통해 메일을 읽습니다.

## 🏗️ 아키텍처 및 개발 전략 (Architecture & Dev Strategies)

이 프로젝트는 품질 향상과 유지보수 효율을 극대화하기 위해 다음 두 가지 아키텍처 설계 수칙을 반드시 따릅니다.

1. **TDD (테스트 주도 개발, Test-Driven Development) 전략:**
   * **테스트 도구:** **Vitest** 및 **React Testing Library (RTL)** 환경을 구축하여 테스트를 수행합니다.
   * **개발 주기:** 반드시 `Red`(실패 테스트 작성) -> `Green`(코드 작성 및 성공) -> `Refactor`(구조 정비) 주기를 따라 핵심 비즈니스 로직을 빌드합니다.
   * 이메일 파서, 자동 분류 조건문, Zustand 스토어 변환 함수 등 핵심 모듈은 무조건 TDD 테스트 커버리지를 먼저 확보합니다.

2. **깊은 모듈 구조 (Deep Module Structure):**
   * 프로젝트 내 프론트엔드 리소스를 얕은 폴더 구조로 흩뿌려 두지 않고, `src/features/` 아래에 비즈니스 도메인 기능별로 격리 개발합니다.
   * 각 기능별 모듈 폴더(예: `src/features/mail-sync/`)는 자체 UI 컴포넌트, 커스텀 훅, Zustand 상태 슬라이스, 그리고 해당 모듈의 TDD 테스트 코드를 포함하여 고도로 응집(Cohesive)되게 설계합니다.
   * 각 모듈은 `index.js` 파일을 통해서만 필요한 구성 요소를 외부에 노출(Export)하며, 다른 모듈의 세부 구현 내용을 직접 참조하지 않도록 차단(Encapsulation)합니다.

## ⚠️ 에이전트 행동 지침 및 제약사항 (Rules of Engagement)

이 프로젝트를 수정하는 모든 AI 에이전트는 다음 수칙을 무조건적으로 준수해야 합니다.

1. **GitHub Flow 개발 전략 준수:**
   * 모든 기능 추가 및 버그 수정은 `main` 브랜치에서 직접 개발하거나 커밋하지 않습니다.
   * 개발을 시작하기 전, 사용자와 조율된 기능 단위를 바탕으로 `feature/기능명` 브랜치를 생성하여 작업해야 합니다.
   * 작업 도중에는 세밀한 단위로 커밋을 쪼개어 작성하며, 커밋 메시지는 한국어로 작성해야 합니다.
   * 구현 완료 후 `main` 브랜치로 바로 합치지 않고, 반드시 Pull Request(PR) 생성 단계를 거쳐 코드 검토 후 머지해야 합니다.

2. **명시적 권한 및 승인 우선 (Permission-First):**
   * 에이전트는 터미널 명령어 실행, 새로운 패키지 설치(`npm install`), 파일 구조 파괴적 변경, 혹은 실제 릴리즈 빌드를 하기 전에 **반드시 사용자에게 구체적인 실행 이유와 명령어 내역을 사전에 설명하고 허락을 구해야 합니다.**
   * 작업을 진행하기 전 **반드시 계획서를 작성해서 사용자에게 공유해야 합니다**
   * 또한 에이전트들이 잘 이해했는지 , 계획이 잘 세워졌는지 사용자가에게 설명을 먼저 한다.
   * 계획서의 위치는 프로젝트 **plan 폴더에 만든다**
   * 계획서의 파일명은 **plan-[기능명].md**로 한다
   * 계획이 완료되면 해당 파일을 삭제한다. 다시 작업을 해야할 경우 재생성한다.
   * 작업을 진행할 경우 어떤 작업을 어떻게 진행하고있는지 설명하는 **새 CMD창을 열어** 서브에이전트로 보여준다.

3. AGENT 연동 확인용 규칙
  * 사용자와 대화할때만 말 끝에 [~용]을 붙인다.
  * [agents.md],[SKILL.md],[Design.md] 파일이 수정이 필요한 경우 **반드시 사용자에게 수정 이유를 먼저 설명한다**
